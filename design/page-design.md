
这个 App 的默认状态不应该像“管理后台”，而应该像“机器人本身”。

用户看到的第一屏不是导航、列表、状态卡片，而是一整个占满横屏的机器人表情。对话、记忆、提醒、设置都属于二级能力，只有用户主动触发时才出现。

这会让产品从“装在机器人里的 App”变成：

> 屏幕就是机器人的脸，其他页面只是它偶尔展开的能力面板。

# 重新定义整体交互结构

整个 App 可以分成两层：

```text
第一层：机器人本体
全屏表情、状态、语音交互、轻提示

第二层：功能空间
对话、记忆、提醒、设置、设备状态
```

默认永远回到第一层。

功能页面打开一段时间没有操作，也可以自动收起，重新回到机器人表情。

# 一、默认首屏：机器人表情页

这是整个产品最重要的页面。

```text
┌──────────────────────────────────────────────────────────┐
│                                                          │
│                                                          │
│                    ◉              ◉                      │
│                                                          │
│                                                          │
│                         ︶                                │
│                                                          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

不显示：

* 导航栏
* 顶部状态栏
* 电量
* 摄像头状态
* 对话输入框
* 最近记忆
* 提醒列表

默认状态下，它就是一张脸。

## 表情承担状态表达

机器人不需要用文字告诉用户“我正在监听”，而是通过眼睛、嘴型、动画和声音反馈状态。

### 待机

```text
眼睛自然眨动
视线偶尔轻微移动
表情放松
```

### 发现主人靠近

```text
眼睛看向用户方向
轻微睁大
身体或屏幕朝向主人
```

### 听到唤醒词

```text
眼睛迅速聚焦
轻微抬眉
发出很短的确认音
```

### 正在听

```text
眼睛保持专注
嘴部或下方出现轻微呼吸波纹
```

### 正在思考

```text
眼睛看向一侧
偶尔眨眼
出现极轻的思考动态
```

### 记住了

```text
眼睛弯起
短暂微笑
可以点头或发出确认音
```

### 不确定

```text
眼睛轻微倾斜
出现疑惑感
随后主动追问
```

### 正在提醒

```text
表情变得认真
眼睛保持打开
提醒结束后恢复正常状态
```

# 二、如何进入其他页面

功能页面不应该常驻，而应该从机器人脸上“展开”。

推荐保留三种入口。

## 方式一：点击机器人

用户单击屏幕，机器人表情轻微缩小，浮出快捷菜单。

```text
┌──────────────────────────────────────────────────────────┐
│                                                          │
│                    ◉          ◉                          │
│                                                          │
│                    今天需要我做什么？                     │
│                                                          │
│       [ 对话 ]      [ 记忆 ]      [ 提醒 ]      [ 设置 ] │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

这个快捷菜单只临时出现。

几秒没有操作后，自动淡出，恢复完整表情。

## 方式二：从屏幕边缘滑出

从左侧向右滑，打开完整功能导航。

```text
┌──────────────┬───────────────────────────────────────────┐
│              │                                           │
│  对话        │                                           │
│  记忆        │              当前功能页面                  │
│  提醒        │                                           │
│  设置        │                                           │
│              │                                           │
│  返回机器人  │                                           │
└──────────────┴───────────────────────────────────────────┘
```

这种方式适合熟悉产品后的用户。

## 方式三：语音打开

用户可以直接说：

```text
“打开记忆”
“看看我今天记住了什么”
“进入设置”
“查看下午的提醒”
```

机器人听懂后，表情缩小并进入对应页面。

这是最符合产品定位的方式。

# 三、功能页面的出现方式

进入功能页时，不建议直接把机器人完全隐藏。

可以让机器人保留成一个小型头像或悬浮球，存在于页面角落。

```text
┌──────────────────────────────────────────────────────────┐
│ 记忆                                             [ ◉ ◉ ] │
│                                                          │
│ 今天                                                     │
│                                                          │
│ ┌──────────────────────────────────────────────────────┐ │
│ │ 钥匙在书桌键盘左侧                                  │ │
│ │ 12:16 · 语音 + 视觉                                 │ │
│ └──────────────────────────────────────────────────────┘ │
│                                                          │
│ ┌──────────────────────────────────────────────────────┐ │
│ │ AirPods 放在显示器后面                              │ │
│ │ 昨天 · 语音                                         │ │
│ └──────────────────────────────────────────────────────┘ │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

右上角的小机器人仍然会：

* 眨眼
* 看向当前选择的内容
* 在说话时有状态变化
* 点击后回到全屏机器人

这样进入工具页面后，用户仍然感觉自己是在和机器人交互，而不是离开机器人进入另一个 App。

# 四、功能页不再采用常驻导航

之前的左侧常驻导航不适合作为全局结构。

更适合改成以下形式：

## 页面内导航

进入功能空间后，才出现顶部或者侧边导航。

```text
[ 对话 ] [ 记忆 ] [ 提醒 ] [ 设置 ]
```

或者：

```text
☰   记忆中心                              [机器人头像]
```

导航只属于功能空间，不属于机器人首页。

## 退出功能页

可以通过这些方式返回：

* 点击右上角机器人头像
* 从右向左滑动
* 点击“返回机器人”
* 说“回去”
* 长时间无操作自动返回

返回过程最好有明显的形态变化：

```text
功能页面逐渐缩小
机器人眼睛从背景中出现
最后恢复全屏表情
```

而不是普通页面跳转。

# 五、重新设计后的页面层级

完整信息架构可以变成：

```text
机器人表情页
│
├── 临时快捷面板
│   ├── 对话
│   ├── 记忆
│   ├── 提醒
│   └── 设置
│
├── 对话空间
│   ├── 当前对话
│   ├── 历史对话
│   └── 回答依据
│
├── 记忆空间
│   ├── 记忆搜索
│   ├── 记忆列表
│   ├── 记忆详情
│   └── 编辑记忆
│
├── 提醒空间
│   ├── 今日提醒
│   ├── 日历事件
│   └── 关联上下文
│
└── 设置空间
    ├── 主人身份
    ├── 语音
    ├── 视觉感知
    ├── 记忆
    └── 设备状态
```

# 六、首屏上的信息应该非常克制

首屏可以显示信息，但必须是短暂出现，而不是固定 UI。

## 唤醒后显示字幕

```text
                    ◉          ◉

                      我在听

           “我把钥匙放在书桌左边了”
```

字幕在识别结束后自动消失。

## 记忆成功

```text
                    ◠          ◠

                     记住了

                钥匙在书桌左边
```

两三秒后只剩机器人表情。

## 即将发生的提醒

```text
                    ◉          ◉

              你 15 分钟后有一个会议

                  [查看详情] [知道了]
```

提醒属于覆盖在脸上的临时卡片，而不是长期存在的右侧栏。

## 环境感知结果

正常情况下不要一直展示“我看到了什么”。

只有发生相关事件时才出现：

```text
我看到你把一串钥匙放在键盘旁边。
需要我记住吗？

[记住] [不用]
```

这样更像机器人在主动观察，而不是监控仪表盘。

# 七、对话页也不应该太像 ChatGPT

当用户主动进入“对话”页面时，可以看到完整对话，但仍然要保留机器人感。

```text
┌──────────────────────────────────────────────────────────┐
│ 对话                                             [◉  ◉]  │
│                                                          │
│             你把钥匙放在书桌键盘左侧。                   │
│             我当时还保存了一张图片。                     │
│                                                          │
│        ┌──────────────────────────────────┐              │
│        │          当时的证据图片           │              │
│        └──────────────────────────────────┘              │
│                                                          │
│                    [ 按住说话 ]                           │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

横屏下甚至可以不采用传统的左右聊天气泡，而是：

* 机器人回答居中展示
* 用户原话弱化展示
* 关键证据作为视觉主体
* 语音是主要输入
* 键盘输入是次要能力

# 八、记忆页面可以更像“机器人脑内”

从机器人表情进入记忆页时，可以有一个过渡：

```text
机器人闭眼
画面进入“记忆空间”
记忆卡片逐渐出现
```

记忆页可以设计得不像文件管理器，而像一组由时间组织的记忆片段。

```text
┌──────────────────────────────────────────────────────────┐
│ 我的记忆                                         [◉  ◉]  │
│                                                          │
│     今天                                                 │
│                                                          │
│     ┌────────────────────┐  ┌────────────────────┐      │
│     │ 钥匙在键盘左侧     │  │ 下午两点有会议     │      │
│     │ [证据图片]          │  │ 关联 3 条记忆      │      │
│     └────────────────────┘  └────────────────────┘      │
│                                                          │
│     昨天                                                 │
│                                                          │
│     ┌────────────────────┐  ┌────────────────────┐      │
│     │ AirPods 在显示器后 │  │ 喜欢冰美式不加糖   │      │
│     └────────────────────┘  └────────────────────┘      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

相比传统列表，这种卡片式记忆空间更有“浏览机器人记忆”的感觉。

# 九、机器人首页仍然要保留隐藏手势

因为全屏都是脸，必须避免用户不知道如何操作。

首次启动时可以出现一次引导：

```text
点击我，可以打开功能
向右滑动，可以查看记忆和设置
也可以直接对我说：“打开记忆”
```

之后不再频繁提示。

可以在屏幕最底部留一个极弱的视觉暗示：

```text
                       ━
```

类似手机底部手势条，但不要显示文字。

# 十、核心设计原则

这套产品的页面设计顺序应该是：

```text
机器人存在感
    ↓
自然语音交互
    ↓
临时信息反馈
    ↓
功能页面
    ↓
管理与设置
```

而不是：

```text
导航
    ↓
页面
    ↓
功能
    ↓
机器人头像
```

最重要的区别是：

> 机器人不是 App 中的一个角色，机器人本身就是 App；页面只是它展开给用户看的记忆、能力和内部状态。

因此默认首屏应当完全由机器人表情占据。只有当用户主动点击、滑动或语音请求时，功能页面才从机器人背后展开。
